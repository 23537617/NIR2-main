# NIR Project: TaskDocument Chaincode (TypeScript)

Добро пожаловать в репозиторий смарт-контракта (чейнкода) `taskdocument` для сети Hyperledger Fabric, переписанного на **TypeScript**.

Этот чейнкод отвечает за управление документами-задачами (Task Documents) внутри распределенного реестра (Ledger) и реализует строгую бизнес-логику для хранения, проверки и отслеживания состояния задач в защищенной блокчейн-сети.

## 🏗 Архитектура разработки

Чейнкод написан с использованием официального и самого современного **TypeScript SDK** (`fabric-contract-api` и `fabric-shim`), что обеспечивает:
- **Строгую типизацию**: Все структуры данных проверяются на этапе компиляции.
- **Декларативное описание транзакций**: Классы смарт-контрактов размечены декораторами `@Info`, `@Transaction`, `@Returns()`.
- **Минимум ошибок**: Исключены проблемы с несовместимостью, характерные для устаревших или неофициальных библиотек на других языках.

#### 📦 Развертывание (Chaincode-As-A-Service)
Для максимальной стабильности, легкости отладки и обновления, этот чейнкод развертывается в сети Fabric не как встроенный (внутри пира), а как внешний сервис (**CCAAS**).
Скрипт `deploy_chaincode.py` конфигурационно привязывает смарт-контракт на пирах к Docker-контейнеру `chaincode-server`, который работает с этим TypeScript-кодом.

## ✨ Реализованный функционал (Advanced Features)

Мы не просто реализовали CRUD, но задействовали весь мощный функционал Hyperledger Fabric:

1. 🛡 **Client Identity (Идентификация клиента)**
   Каждое создание задачи автоматически определяет и записывает создателя (MSP ID и данные сертификата X.509) с помощью `ctx.clientIdentity`.
2. 📜 **History API (Слепок истории транзакций)**
   Метод `get_task_history(task_id)` позволяет получить весь путь жизненного цикла конкретной задачи от момента её создания до последнего изменения статуса. Возвращаются точные метки времени (`tx_timestamp`) и идентификаторы транзакций.
3. 🔍 **Rich Query (Предметно-реляционные запросы)**
   Поскольку сеть использует CouchDB, чейнкод поддерживает "богатые запросы". Метод `query_tasks` принимает JSON-селекторы CouchDB (например, `{"selector":{"docType":"task","assignee":"Ivanov"}}`), позволяя находить задачи не только по ключу, но и по значениям полей.
4. 📑 **Pagination (Пагинация выборок)**
   Для работы с большими ответами `query_tasks` поддерживает безопасную пагинацию (передаются аргументы `page_size` и `bookmark`). Запросы не зависают, а возвращают новую `bookmark` для получения следующей страницы.
5. 🔔 **Events (События чейнкода)**
   Каждая мутирующая транзакция (например, `create_task`, `update_task_status`, `add_document_version`) генерирует блокчейн-событие (`onTaskCreated`, `onTaskStatusUpdated`). Внешние приложения (Клиенты/Frontend) могут подписываться на эти сигналы для мгновенного обновления UI.
6. ⏱ **Детерминированность (Отсутствие расхождений состояний)**
   В смарт-контракте не используется системное время (`new Date()`). Вместо него используется консенсусное время транзакции из Fabric (`ctx.stub.getTxTimestamp()`), гарантирующее, что все пиры придут к одинаковому результату (Read-Write Set).

## 🚀 Деплой и запуск

Поскольку чейнкод использует архитектуру **CCAAS**, его запуск состоит из двух компонентов: упаковки (для Fabric) и запуска контейнера (для логики).

### 1. Запуск контейнера чейнкода (сервера TypeScript)
В текущей директории с чейнкодом выполните сборку и запуск Docker-контейнера через Docker Compose:
```bash
docker compose -f docker-compose.chaincode.yaml up -d --build chaincode-server
```
Это соберет TypeScript проект (на базе `node:18-alpine`) и поднимет легкий сервер, слушающий порт `9999`.

### 2. Регистрация в сети Fabric
Если сеть чистая (только что подняли `network_setup.py up`), вам нужно "объяснить" пирам, где находится сервер чейнкода. Для этого используйте готовую утилиту:
```bash
python deploy_chaincode.py
```
Скрипт создаст архив с `connection.json`, установит его на все пиры в обоих организациях и произведет коммит (Commit) чейнкода в канал.

*Примечание: Если вы изменили только логику (`taskContract.ts`), вам не нужно перезапускать `deploy_chaincode.py`. Достаточно просто перезапустить докер-контейнер `chaincode-server`, и сеть Fabric автоматически "подцепит" новую логику на лету!*

## 🛠 Структура проекта (папка `chaincode-ts`)

- `src/index.ts`: Главный входной файл чейнкода. Экспортирует контракты.
- `src/models.ts`: TypeScript-интерфейсы `Task` и `DocumentVersion`, описывающие структуру данных, сохраняемых в Ledger.
- `src/taskContract.ts`: Вся логика выполнения транзакций. Сердце смарт-контракта.
- `Dockerfile`: Мульти-этапная сборка контейнера. Компилирует TypeScript в JavaScript и копирует только необходимое (production dependencies).
- `package.json`: Зависимости, скрипты сборки (`build`) и команда старта `start`, которая запускает `fabric-chaincode-node`.

## 📄 Основные транзакции (API)

* `create_task(task_id, title, description, assignee)` — Создает новую задачу, назначая ей статус "NEW".
* `get_task(task_id)` — Возвращает состояние задачи по ID.
* `update_task_status(task_id, new_status, updated_by)` — Изменяет статус задачи и генерирует событие.
* `get_task_history(task_id)` — Возвращает список всех предшествующих изменений задачи.
* `query_tasks(query_string, page_size, bookmark)` — Поиск задач с использованием запросов CouchDB.
* `add_document_version(task_id, document_id, version, content_hash, uploaded_by, metadata)` — Добавляет ссылку на версию документа (расположенного в IPFS или БД) с привязкой к конкретной задаче.
* `get_document_versions(task_id, document_id)` — Возвращает историю версий конкретного документа.
